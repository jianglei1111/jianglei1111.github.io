# EAGER 论文工作简要说明

更新时间：2026-07-21

## 1. 研究背景

本项目研究开放域多跳问答中的检索增强生成（multi-hop RAG）。这类问题通常不能依靠
一次检索直接回答，而是需要先找到一段证据，从中识别新的实体或关系，再继续检索后续
证据，最终组合成完整答案。

现有方法主要有两个局限。一方面，只围绕原问题进行全局相似度检索，容易漏掉必须通过
中间实体才能找到的后续证据；另一方面，预先生成固定的子问题或推理路径，一旦拆解或
中间答案出错，错误就会沿整条链传播。

本文将多跳问答重新建模为一个 **query-time evidence-state construction** 问题，并
提出 EAGER。系统在检索过程中维护一个不断更新的证据状态：当前证据暴露新的实体锚点，
step-level signal 指明下一步需要寻找什么，系统再从实体邻域和语义候选中筛选证据，
直到形成足以回答原问题的证据集合。

## 2. 系统总体架构

![EAGER 系统总体架构](images/updates/eager/eager-overview.png)

*图 1：EAGER 系统总体架构。Retrieval anchors 决定候选证据在哪里扩展，filtering
signals 指导扩展候选池内的分层筛选，保留的 evidence state 在多轮检索中持续更新，
最终与原问题共同用于答案生成。*

系统包含三个关键环节：

1. **候选可达性**：从原问题和已保留证据中提取实体、标题等锚点，激活相关实体邻域，
   并与语义检索候选合并；
2. **信号条件选择**：RQ 或 SC 给出当前步骤的检索目标，系统先选择相关段落，再使用
   reranker 选择句子级证据；
3. **证据状态更新**：选中的证据加入 evidence state，同时暴露下一轮可用的新锚点；
   最终答案始终由原问题和累计证据共同生成。

本文固定研究两个完整系统，不使用逐题路由器：

- **EAGER-RQ** 根据原问题和当前证据，在线生成下一步 retrieval question；
- **EAGER-SC** 预先生成 answer-dependent subquestion chain，并使用前序中间答案
  物化后续子问题。

RQ 更适合根据当前证据灵活调整方向；SC 的依赖结构更明确，在长链和顺序依赖问题上
可能更稳定。论文通过事后证据结构分析研究二者的适用边界，而不是在测试时选择最优模式。

## 3. 论文核心贡献

### 贡献一：查询时证据状态构造

本文将多跳 RAG 建模为查询时证据状态构造，显式区分候选证据可达性、信号条件选择和
状态转移。后续检索可以利用已有证据暴露的新实体，而不完全依赖原问题或一次性拆解。

### 贡献二：信号条件下的分层证据选择

本文提出统一的 signal-conditioned hierarchical evidence selection，并使用 online RQ
和 answer-dependent SC 两种控制方式实例化。两种信号共享同一检索、重排和答案生成
系统，因此可以公平比较其答案准确率、支持链覆盖和推理成本。

### 贡献三：有界实体邻域激活

本文使用问题和已有证据中的锚点激活实体邻域，并在候选进入 coarse scoring 和
reranking 前施加固定预算。在 anchor 与 pre-score candidate budget 固定时，每一步
在线处理的候选规模与总语料规模解耦。该贡献强调大规模语料下的候选空间控制，不声称
实体扩展必然提高所有数据集的准确率，也不声称整个系统严格为常数时间。

## 4. 当前核心实验结果

正式主实验覆盖 HotpotQA、2WikiMultiHopQA 和 MuSiQue，每个数据集 1,000 道题。
EAGER-RQ 和 EAGER-SC 分别完整运行，共得到 6,000 道主系统预测。论文使用 ConAcc
和 GPTAcc 作为答案指标。

### 4.1 主结果

表中每个单元格为 ConAcc/GPTAcc。

| 方法 | HotpotQA | 2Wiki | MuSiQue |
| --- | ---: | ---: | ---: |
| GPT-4o-mini Zero-shot | 35.9 / 32.9 | 36.9 / 24.2 | 13.3 / 13.9 |
| Dense Top-5 + Qwen rerank | 53.5 / 55.2 | 42.7 / 36.6 | 27.5 / 30.5 |
| IRCoT-style Top-5 | 63.1 / 68.9 | 67.2 / 62.7 | 34.9 / 38.3 |
| LinearRAG MiniLM | 65.4 / 65.4 | 70.2 / 64.1 | 32.3 / 36.2 |
| **EAGER-RQ** | **65.7 / 71.5** | **76.6 / 72.0** | **38.3 / 39.6** |
| **EAGER-SC** | **66.4 / 71.7** | **73.6 / 69.6** | **41.7 / 43.5** |

EAGER 的两个固定变体在三个数据集上都具有竞争力。HotpotQA 上 RQ 与 SC 基本持平；
2Wiki 上 RQ 最好，相对 LinearRAG 提高 6.4 ConAcc 和 7.9 GPTAcc；MuSiQue 上 SC
最好，相对 IRCoT-style 提高 6.8 ConAcc 和 5.2 GPTAcc。

### 4.2 支持链覆盖

| 数据集 | RQ Candidate/Final Full | SC Candidate/Final Full |
| --- | ---: | ---: |
| HotpotQA | 96.4 / 90.0 | 95.6 / 90.9 |
| 2Wiki | 96.8 / 92.2 | 96.7 / 92.8 |
| MuSiQue | 68.1 / 39.1 | 73.6 / 58.1 |

HotpotQA 和 2Wiki 上，两个系统都能在候选阶段覆盖绝大多数完整支持链，并在最终证据
状态中保留约 90% 以上。MuSiQue 的长依赖和分散证据更困难，SC 的 Final Full 为
58.1%，明显高于 RQ 的 39.1%，与 SC 在该数据集上的答案优势一致。

### 4.3 机制消融

| 系统 | HotpotQA | 2Wiki | MuSiQue |
| --- | ---: | ---: | ---: |
| EAGER-NoSignal | 63.9 / 70.0 | 67.6 / 63.4 | 32.9 / 35.5 |
| Full RQ | 65.7 / 71.5 | 76.6 / 72.0 | 38.3 / 39.6 |
| Full SC | 66.4 / 71.7 | 73.6 / 69.6 | 41.7 / 43.5 |

相对 NoSignal，RQ 在三个数据集上分别提高 1.8/1.5、9.0/8.6 和 5.4/4.1 个百分点；
SC 分别提高 2.5/1.7、6.0/6.2 和 8.8/8.0 个百分点。这说明实体邻域主要解决“证据
在哪里”，step-level signal 则进一步解决“当前应该选择什么证据”。

更细的消融也显示，hierarchical refinement 和 evidence-derived state anchors 并非在
所有数据集上单调增益。它们可能改善局部选择，也可能在长链问题中造成过早剪枝或
evidence drift。因此论文将其解释为与证据结构有关的机制，而不是普遍有效的独立模块。

### 4.4 百万级规模结果

规模实验将三个数据集从原始数万级 evidence units 扩展到约 100 万。为控制实体邻域
在 coarse scoring 前的候选枚举，固定使用 `pre_score_candidate_limit=200`。前 20 题
用于工程 pilot，参数冻结后使用其余 80 题进行数万和百万端点比较。

| 数据集 | Hard Local 数万 -> 百万 | 百万/数万 | 百万档相对原 Local 加速 |
| --- | ---: | ---: | ---: |
| HotpotQA | 7.94 -> 8.06 s | 1.02x | 3.93x |
| 2Wiki | 20.33 -> 16.63 s | 0.82x | 5.69x |
| MuSiQue | 22.16 -> 23.67 s | 1.07x | 6.06x |

语料规模增加约 23--45 倍后，完整 RQ 检索延迟只变化 0.82--1.07 倍，每一步 expanded
candidate 最大值均不超过 200。相对原 Local，百万档加速 3.93--6.06 倍；同题
ConAcc/GPTAcc 差异均未达到统计显著，没有检测到由硬预算造成的答案准确率下降。

这一结果支持“固定预算下在线候选处理规模与总语料规模解耦”，但不能写成整个系统严格
`O(1)`。Hard Local 目前仍比 Global ANN 慢约 1.13--2.26 倍，其优势是保留实体局部
证据可达性并使成本更加稳定，而不是绝对速度优于 ANN。
