# TRACER：面向证据状态的成本感知 RAG 路由

更新时间：2026-07-21

## 项目简介

TRACER 研究的是一个很实际的 RAG 问题：**当系统已经拿到一批检索证据后，它应该直接回答，还是继续检索、处理时间问题，或者先解决证据冲突？**

普通 RAG 往往对所有问题执行相同流程，例如固定检索 Top-5，再把全部文档交给大模型。但相同数量的检索结果可能处于完全不同的状态：

- 有些证据已经足够，可以直接回答；
- 有些证据缺少关键的一跳，需要继续检索；
- 有些证据描述的是旧状态，不能用于回答当前问题；
- 有些证据在同一时间条件下互相冲突，需要进一步验证。

这些情况需要的操作不同。继续增加文档不一定能够解决问题，还可能增加 Token、噪声和错误回答风险。

TRACER 因此不把路由理解成简单的“检索或不检索”，而是先判断当前证据出了什么问题，再选择与该问题相匹配的处理动作。

```text
用户问题 + 初始证据
          ↓
判断证据是否缺失、过时或冲突
          ↓
估计不同动作的正确率和成本
          ↓
选择直接回答、补充检索、时间检索或冲突验证
          ↓
生成有证据支持的最终答案
```

## 核心工作

### 1. 从“是否检索”转向“证据状态—动作”路由

TRACER 关注三个主要证据问题：

| 状态 | 含义 | 对应处理方式 |
| --- | --- | --- |
| 证据缺失 | 当前材料没有覆盖回答所需的全部事实 | 补充检索 |
| 时间不匹配 | 检索到了相关事实，但事实版本不符合问题要求的时间 | 时间感知检索 |
| 证据冲突 | 与问题相关的证据在相同条件下支持不同结论 | 冲突验证 |

系统不会把三个状态强制压缩成一个类别。同一个问题可以同时存在证据缺失和时间不匹配，因此模型输出的是多个状态概率。

在状态判断之后，系统分别估计每个动作执行后得到正确答案的概率以及需要消耗的 Token。这样可以在正确率和成本之间选择合适的工作点，而不是默认执行最昂贵的流程。

### 2. 用 Fact-Time-Source 描述证据

仅靠文本相似度很难判断一条证据是否真正有用。TRACER 进一步考虑三类信息：

- **Fact**：证据表达了什么事实，与问题需要的事实是否相关；
- **Time**：事实在哪个时间范围内有效，是否适用于当前问题；
- **Source**：证据来自哪里，不同证据是相互支持、互相冲突，还是新版本取代旧版本。

当前模型在三个层次上处理这些信息：

```text
单条证据：是否与问题相关
证据对：支持 / 冲突 / 版本替代
整道问题：证据缺失 / 时间不匹配 / 冲突
```

这种表示的意义在于，系统不仅知道“整题可能有冲突”，还能够利用具体证据之间的关系形成判断，为后面的时间过滤和冲突处理提供依据。

### 3. 建立可比较的全动作实验环境

TRACER-Bench v1 不把某个动作永久标记为唯一正确答案。项目会在同一问题和同一实验环境下分别运行所有候选动作，记录每个动作的答案、证据、Token 和调用情况，再根据实际结果计算最优动作。

这种设计能够区分两个问题：

- Router 有没有选中真正有效的动作；
- 即使多个动作都能答对，Router 有没有选择成本更低的那个。

数据同时保留证据原文、来源、时间、原始捕获和哈希，实验划分保证同一事件、近重复问题和共享核心证据不会跨越训练集与测试集。

## 数据与实验环境

TRACER-Bench v1 当前包含：

| 项目 | 数量 |
| --- | ---: |
| 基础问题 | 225 |
| 不同证据状态形成的样本 | 1,192 |
| 多跳 / 单跳问题 | 146 / 79 |
| 数据来源 | FreshQA、MQuAKE-T、SealQA、TimeQA |
| 固定分组 folds | 5 |
| 事实级可用问题 | 209 |

基础实验使用 `gpt-4o-mini-2024-07-18` 作为回答模型，使用确定性 BM25 作为检索器。当前执行的四个动作是：

- `answer`：使用现有证据直接回答；
- `retrieve`：使用 BM25 补充证据；
- `temporal_retrieve`：检索后根据事实有效时间进行重排；
- `verify`：针对冲突双方补充证据，通过独立来源支持情况选择并清理错误一侧。

除了受控证据状态，项目还对 225 个正常问题重新执行了真实 BM25 Top-5，并逐条复核其中 209 个事实级样本。正常检索结果中有 46 个主要缺失关键证据，29 个主要使用了过时证据，说明路由问题并不只存在于人工构造的变体中。

## 目前得到的主要结果

### 不同证据状态确实需要不同动作

固定动作在全部 1,192 个样本上的总体表现如下：

| 动作 | EM | Grounded EM | 平均 Token |
| --- | ---: | ---: | ---: |
| 直接回答 | 56.5% | 45.9% | 1,868.6 |
| 普通检索 | 70.9% | **67.9%** | 3,078.0 |
| 时间检索 | **71.7%** | **67.9%** | 3,006.1 |
| 冲突验证 | 71.3% | 66.8% | 3,372.5 |

只看总体平均值时，几个检索动作差距不大；按证据状态拆开后，差异非常清楚：

| 证据状态 | 直接回答 EM | 最佳动作 | 最佳动作 EM |
| --- | ---: | --- | ---: |
| 缺失关键证据 | 63.0% | 普通检索 / 时间检索 | 71.9% |
| 只有过时证据 | 16.9% | 时间检索 | 72.0% |
| 受控证据冲突 | 32.4% | 冲突验证 | 62.7% |

在受控冲突上，Verify 比普通检索高 `8.0` 个百分点 EM，95% 置信区间为 `[+1.3, +14.7]`。这说明冲突验证不是普通增加 Top-k 的替代写法，而是具有独立价值的处理动作。

### 路由器已经能够识别主要证据问题

当前多标签状态模型在固定五折上的结果为：

| 状态 | ROC-AUC |
| --- | ---: |
| 证据缺失 | 0.846 |
| 时间不匹配 | 0.880 |
| 证据冲突 | **0.893** |

这说明仅使用动作执行前能够观察到的问题和证据信息，已经可以较稳定地区分三种主要状态。状态概率会继续作为动作成功率模型的输入，而不是直接通过人工规则转换成动作。

### 当前路由能够减少一部分成本

策略层会预测四个动作的成功概率和 Token 成本，并在达到成功率阈值的动作中选择成本较低的动作。

| 策略 | EM | Grounded EM | 平均 Token |
| --- | ---: | ---: | ---: |
| 固定时间检索 | **72.65%** | **68.76%** | 3,006.1 |
| 路由阈值 0.8 | 70.89% | 66.25% | 2,611.0 |
| 路由阈值 0.9 | 71.22% | 66.70% | 2,670.4 |
| 动态 Oracle | 79.03% | 76.21% | 2,213.8 |

阈值 `0.9` 相比固定时间检索平均减少约 `336` Token，EM 下降约 `1.4` 个百分点。当前结果体现的是可调节的质量—成本折中，还不是在所有指标上全面超过固定 RAG。

在另外一组正常 BM25 Top-5 实验中，Evidence-aware Router 达到 `75.56%` EM 和 `2,456.1` 平均 Token。它与固定时间检索的 EM 相同，但平均减少约 `262` Token，说明路由信号能够从受控证据状态迁移到正常检索结果。

### Fact-Time-Source 关系对状态判断有明显帮助

在五折可解释基线中，加入时间和来源特征后，Case、Span 和 Pair 三个任务的 Macro-F1 分别提高 `0.150`、`0.148` 和 `0.066`。

进一步的神经模型实验得到：

| 模型 | Case AUC | Span AUC | Pair AUC |
| --- | ---: | ---: | ---: |
| V2 Multi-view FTS | 0.651 | 0.696 | 0.806 |
| V3.0 Frozen Relation Probe | **0.872** | - | - |
| V3.1 Detached | 0.865 | 0.697 | **0.795** |
| V3.1 Joint | 0.863 | **0.703** | 0.784 |

V3.0 使用冻结的证据相关性和证据关系概率，将 Case AUC 从 raw case 基线的 `0.824` 提高到 `0.872`；相对只保留相同证据数量和连接结构的对照也提高了 `0.062`。这表明真正有价值的是证据之间的关系，而不只是证据更多或模型更大。

V3.1 已经让整题状态损失能够反向更新证据相关性和关系模块，但没有进一步超过冻结模型，并且 Pair AUC 略有下降。因此目前保留的结论很克制：**Fact-Time-Source 关系有效，但在当前数据规模下，增加联合训练复杂度没有带来额外收益。**

## 当前进展

目前已经完成的部分包括：

- TRACER-Bench v1 的构建、来源审计、全局去重和最终验证；
- 四个动作的执行器、缓存、逐题日志、答案评分和完整动作结果矩阵；
- 基于 GPT-4o-mini 和轻量模型的 Router 基线；
- 证据状态、动作成功率和 Token 成本预测；
- 成本感知阈值策略和正常 BM25 Top-5 回放；
- Fact-Time-Source 的可解释基线、Multi-view 模型和关系聚合模型。

项目现在已经形成了从“问题和初始证据”到“状态判断、动作预测和动作执行”的完整实验闭环。最有说服力的发现不是某个复杂模型刷新了一个单独指标，而是下面这条完整证据链：

> 不同证据问题需要不同动作；这些问题可以从在线证据中识别；状态和动作收益能够用于减少不必要的处理成本；显式事实、时间和来源关系能够进一步改善状态判断。

当前仍需谨慎看待三个边界：Verify 的主要定量结果来自受控冲突；P4 神经模型目前是单 fold pilot；所有主要系统结果仍基于同一个生成模型和 BM25 环境。因此，这些结果已经足以说明研究方向成立，但还不能作为跨模型、跨检索器泛化的最终结论。
